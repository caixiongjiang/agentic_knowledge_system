# 回收站删除与恢复交互设计

## 1. 设计方案概述

本系统采用**整体入站 + 原子化操作**的回收站设计方案：

- 删除文件夹时，**整棵文件夹树**作为一个整体移入回收站，而非将文件散开
- 回收站内文件夹可以展开浏览（只读预览），但**恢复和永久删除只能对顶层条目操作**
- 单独删除的文件作为独立条目出现在回收站顶层

该方案与 macOS Finder、Windows 回收站、Google Drive 等主流产品的交互一致。

---

## 2. 核心数据模型

### `deleted` 字段三态

| 值 | 含义 | 回收站顶层可见 | 可独立操作 |
|---|---|---|---|
| **0** | 正常（未删除） | 否 | - |
| **1** | 用户直接删除的条目 | **是** | **是** |
| **2** | 级联删除（被父文件夹连带删除） | **否** | **否（只读）** |

`deleted=1` 和 `deleted=2` 的区分是交互设计的核心：

- `deleted=1`：用户主动执行了删除操作的直接目标，在回收站顶层可见，可恢复或永久删除
- `deleted=2`：因为父文件夹被删除而级联标记的子项，仅在展开父文件夹后可见，不提供独立操作

---

## 3. 删除行为

### 3.1 删除单个文件

```
操作：删除文件 report.pdf
结果：report.pdf → deleted=1

回收站顶层：
└── 📄 report.pdf（deleted=1）
```

### 3.2 删除文件夹（含文件）

整棵文件夹树 + 所有文件一起移入回收站：

```
操作：删除文件夹 项目A/

结果：
  项目A/         → deleted=1（顶层可见）
  项目A/子目录1/  → deleted=2（级联）
  项目A/子目录2/  → deleted=2（级联）
  项目A/子目录1/文件1.txt → deleted=2（级联）
  项目A/子目录1/文件2.txt → deleted=2（级联）
  项目A/README.md         → deleted=2（级联）

回收站顶层：
└── 📁 项目A（deleted=1，可展开浏览）
```

### 3.3 删除空文件夹

直接物理删除，**不进回收站**（空文件夹没有恢复价值）。

### 3.4 删除行为汇总

| 场景 | 处理方式 | 进回收站 |
|---|---|---|
| 删除单个文件 | 文件 `deleted=1` | 是 |
| 批量删除文件 | 每个文件 `deleted=1` | 是 |
| 删除含文件的文件夹 | 目标文件夹 `deleted=1`，后代 `deleted=2`，文件 `deleted=2` | 是 |
| 删除空文件夹 | 直接物理删除 | 否 |

## 4. 回收站浏览

### 4.1 顶层列表

`GET /api/knowledge/trash/list`

返回所有 `deleted=1` 的条目（文件夹和文件混合列表），建议按删除时间倒序排列。

```
回收站顶层：
├── 📁 项目A（deleted=1，2 分钟前）
├── 📄 report.pdf（deleted=1，1 小时前）
├── 📁 旧笔记（deleted=1，3 天前）
└── 📄 草稿.docx（deleted=1，1 周前）
```

支持按 `knowledge_base_id` 筛选。

### 4.2 展开文件夹（只读预览）

- `GET /api/knowledge/trash/folder/{folder_id}/children` — 子文件夹列表
- `GET /api/knowledge/trash/folder/{folder_id}/files` — 子文件列表

用户点击回收站内的文件夹，可以逐级展开查看内部结构：

```
📁 项目A（展开后）
├── 📁 子目录1
│   ├── 📄 文件1.txt
│   └── 📄 文件2.txt
├── 📁 子目录2
└── 📄 README.md
```

> **重要**：展开后的子项（deleted=2）**仅供预览，不提供"恢复"或"永久删除"按钮**。
> 用户通过浏览了解文件夹内有什么内容，然后对顶层条目（整个文件夹）做决策。

## 5. 恢复行为

### 5.1 恢复文件夹（原子操作）

`POST /api/knowledge/trash/restore/folder/{folder_id}`

**仅允许对 `deleted=1` 的文件夹执行。** 一次性恢复整棵文件夹树：

- 目标文件夹 + 所有后代文件夹 → `deleted=0`
- 所有关联文件 → `deleted=0`

```
操作前（回收站）：          操作后（正常目录）：
📁 项目A                  📁 项目A
├── 📁 子目录1              ├── 📁 子目录1
│   ├── 📄 文件1.txt        │   ├── 📄 文件1.txt
│   └── 📄 文件2.txt        │   └── 📄 文件2.txt
└── 📄 README.md           └── 📄 README.md
```

### 5.2 恢复单个文件

`POST /api/knowledge/trash/restore/file/{file_id}`

**仅允许对 `deleted=1` 的文件执行**（即用户直接删除的散文件）。

- 文件 → `deleted=0`
- 如果其父文件夹已不存在，将文件移到根目录

### 5.3 不允许的操作

| 操作 | 是否允许 | 原因 |
|---|---|---|
| 恢复顶层文件夹（deleted=1） | ✅ 允许 | 原子化恢复整棵树 |
| 恢复顶层文件（deleted=1） | ✅ 允许 | 用户直接删除的散文件 |
| 恢复文件夹内的子文件夹（deleted=2） | ❌ 不允许 | 会导致幽灵文件问题 |
| 恢复文件夹内的子文件（deleted=2） | ❌ 不允许 | 会导致幽灵文件问题 |

### 5.4 为什么不允许恢复 deleted=2 的子项

假设用户删除了 `A/B/C/`（含 `x.txt`、`y.txt`、`z.txt`），如果允许只恢复子项 `x.txt`：

1. 系统必须同时恢复祖先链 A → B → C（否则 `x.txt` 无处可放）
2. 文件夹 A 从回收站顶层**消失**
3. `y.txt` 和 `z.txt` 仍是 `deleted=2`，但它们的父文件夹 C 已经变成 `deleted=0`
4. `y.txt` 和 `z.txt` 在正常目录和回收站中都不可见 → **幽灵文件**

通过限制"只能操作 deleted=1 的顶层条目"，从根本上避免这个问题。

## 6. 永久删除行为

### 6.1 永久删除文件夹

`DELETE /api/knowledge/trash/folder/{folder_id}`

**仅允许对 `deleted=1` 的文件夹执行。** 级联硬删除：

1. 文件夹 + 所有后代文件夹 → 物理删除
2. 所有关联文件 → 物理删除
3. 级联清理 Milvus 向量数据、MongoDB 文档数据、对象存储文件

### 6.2 永久删除单个文件

`DELETE /api/knowledge/trash/file/{file_id}`

**仅允许对 `deleted=1` 的文件执行。**

### 6.3 清空回收站

`DELETE /api/knowledge/trash/empty`

永久删除回收站中的所有条目（`deleted=1` 和 `deleted=2` 全部硬删除）。

## 7. 完整 API 汇总

### 7.1 正常操作区（触发软删除）

| 方法 | 路径 | 功能 | 说明 |
|---|---|---|---|
| DELETE | `/api/knowledge/folder/{folder_id}` | 删除文件夹 | 含文件 → 回收站；空 → 物理删除 |
| DELETE | `/api/knowledge/folder/file/{file_id}` | 删除文件 | deleted=1，进回收站 |
| POST | `/api/knowledge/folder/file/batch-delete` | 批量删除文件 | 每个 deleted=1 |

### 7.2 回收站区

| 方法 | 路径 | 功能 | 操作对象 |
|---|---|---|---|
| GET | `/api/knowledge/trash/list` | 顶层列表 | deleted=1 |
| GET | `/api/knowledge/trash/folder/{folder_id}/children` | 浏览子文件夹 | 只读预览 |
| GET | `/api/knowledge/trash/folder/{folder_id}/files` | 浏览子文件 | 只读预览 |
| POST | `/api/knowledge/trash/restore/folder/{folder_id}` | 恢复文件夹 | 仅 deleted=1 |
| POST | `/api/knowledge/trash/restore/file/{file_id}` | 恢复文件 | 仅 deleted=1 |
| DELETE | `/api/knowledge/trash/folder/{folder_id}` | 永久删除文件夹 | 仅 deleted=1 |
| DELETE | `/api/knowledge/trash/file/{file_id}` | 永久删除文件 | 仅 deleted=1 |
| DELETE | `/api/knowledge/trash/empty` | 清空回收站 | 全部 |

## 8. 前端交互指南

### 8.1 回收站页面布局

```
┌──────────────────────────────────────────────────┐
│  🗑️ 回收站                    [清空回收站]       │
│                                                  │
│  筛选: [所有知识库 ▾]                             │
│                                                  │
│  ┌──────────────────────────────────────────────┐│
│  │ 📁 项目A文档        2分钟前   [恢复] [删除]  ││
│  │   └─ 展开预览:                               ││
│  │     📁 子目录1                                ││
│  │       📄 文件1.txt                            ││
│  │       📄 文件2.txt                            ││
│  │     📁 子目录2                                ││
│  │     📄 README.md                              ││
│  │                                               ││
│  │ 📄 report.pdf      1小时前   [恢复] [删除]   ││
│  │ 📁 旧笔记           3天前    [恢复] [删除]   ││
│  │ 📄 草稿.docx        1周前    [恢复] [删除]   ││
│  └──────────────────────────────────────────────┘│
└──────────────────────────────────────────────────┘
```

### 8.2 交互规则

| 元素 | 交互 |
|---|---|
| 顶层文件夹 | 可展开预览内容；提供"恢复"和"永久删除"按钮 |
| 顶层文件 | 提供"恢复"和"永久删除"按钮 |
| 展开后的子项 | **仅展示，不提供任何操作按钮** |
| 清空回收站 | 需要二次确认弹窗 |
| 永久删除 | 需要二次确认弹窗 |
| 恢复文件夹 | 建议提示"将恢复 X 个文件夹和 Y 个文件" |

### 8.3 展开预览的目的

用户在决定"恢复还是永久删除"之前，需要知道文件夹里有什么。展开预览让用户能看到：

- 文件夹内有多少个子文件夹和文件
- 具体有哪些文件（文件名、大小、类型）
- 帮助用户做出"要不要恢复整个文件夹"的决策

## 9. 方案对比与选型理由

### 曾考虑的方案

| 方案 | 描述 | 结论 |
|---|---|---|
| A: 散开文件 | 删除文件夹时把文件打散放入回收站顶层 | ❌ 回收站被淹没，无法原子化撤销 |
| B-完整版: 保留层级+子项可操作 | 回收站内可对任意子项恢复/删除 | ❌ 恢复子项会导致幽灵文件 |
| **B-原子化: 保留层级+顶层操作** | 回收站内可浏览但只能操作顶层条目 | ✅ **采用** |

### 选型理由

1. **操作与认知一致**：删除一个文件夹 = 回收站多一个文件夹，恢复一个文件夹 = 全部恢复
2. **无幽灵文件**：不允许部分恢复，deleted 状态始终一致
3. **回收站不被淹没**：文件夹是一个条目，不是散开的 N 个文件
4. **前端实现简单**：展开后只需渲染列表，不需要在子项上放操作按钮
5. **主流产品验证**：macOS Finder、Windows 回收站、Google Drive 均采用类似方案

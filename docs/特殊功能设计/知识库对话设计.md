## 1. 模块定位

`Chat` 是面向终端用户的**知识库问答接口**，向上承接前端对话 UI，向下编排：

1. **检索层**（`src/service/knowledge/retrieve_service.py`）：每条用户输入触发一次**服务端无条件初次召回**，把命中的 `ChunkItem` 注入对话上下文。
2. **LLM 层**（`src/client/llm/`）：负责流式输出、思考链（reasoning）、工具调用（tool calling）。
3. **导航工具层**（`src/service/chat/tools.py::KnowledgeNavToolKit`）：把已有的导航能力（粒度上下钻、上下文窗口、文档骨架）以 OpenAI Function Schema 暴露给 LLM，让模型在多轮中**自主细化**已有上下文。
4. **持久化层**：MySQL 存会话元信息，MongoDB 存消息内容（Beanie ODM）。
5. **传输层**：以 **WebSocket** 为主流式协议向前端推送细粒度事件（思考、正文、工具调用、引用、用量）。

> **本模块不做什么**：不实现任何检索算法（全部委托 `RetrieveService`），也不直接管理底层模型连接（全部委托 `LLMClient` + `[llm.presets.*]`）。

设计原则：

| 原则 | 含义 |
|---|---|
| **薄编排、厚复用** | Chat 自身不实现检索逻辑，全部委托 `RetrieveService` 与 `capabilities/navigation` |
| **流式优先** | 思考、正文、工具调用、引用都通过事件流逐步推送，避免长时间空白 |
| **服务端无条件初次召回** | "是否检索"的决策权收回到服务端，保证最低召回保障；LLM 仅决定"是否进一步细化" |
| **Agent 工具白名单** | 暴露给 LLM 的工具集合在配置里显式声明，不允许动态注入未审计的工具 |
| **Stop / Cancel 一等公民** | 客户端可主动断流；服务端在收到 stop 信号或 WS 断开时立刻 cancel 上游 LLM 调用 |
| **可审计** | 每条 assistant 消息持久化时必带 `tool_calls / citations / finish_reason / usage` |

## 3. 两种工作模式

| 模式 | 触发 | 行为 |
|---|---|---|
| **RAG 单轮**（fast path） | `agent_mode=false` | 服务端先检索一次 → LLM 流式作答（不绑定工具） |
| **Agent 模式**（默认） | `agent_mode=true` | 服务端先检索一次 → LLM 流式生成，可在多轮中自主调用 `expand_context / drill_down / skeleton` 细化上下文 → 触达 `max_tool_rounds` 后强制收尾输出最终答案 |

### 3.1 关键取舍：Agent 模式不暴露"再次召回"工具

Agent 工具白名单**不包含** `retrieve_knowledge` / `re_query` 等"再次召回"型工具。原因：

1. 是否再次召回的决策权应收敛到服务端规则（未来由验证 Agent 兜底），LLM 不直接决定"扩大召回范围"。
2. 与 `ResultValidator` 的设计完全一致，可大幅复用基类代码（`KnowledgeNavToolKit`）。
3. 模型只在已有命中片段的基础上做导航式细化，行为可预测、可审计。

如果要恢复"再次召回"能力，建议沿 `ValidatorToolKit` 的扩展路径：继承 `KnowledgeNavToolKit`，通过 `_register_extra_tools()` / `_extra_tool_schemas()` 钩子追加，而不是直接修改基类。

## 4. WebSocket 协议规范（v1）

子协议常量：`aks-chat-v1`（见 `src/chat/protocol.py::WS_SUBPROTOCOL`）。

### 4.1 握手与鉴权

```
WS /api/chat/ws?token=<user_id>
  Subprotocol: aks-chat-v1[.<token>]
```

鉴权两种方式（任选其一），均在 `accept()` **之前**校验：

| 方式 | 用法 | 适用场景 |
|---|---|---|
| Query token | `?token=<user_id>` 或 `?user_id=<user_id>` | 浏览器原生 WS（不能设 header） |
| Subprotocol | `Sec-WebSocket-Protocol: aks-chat-v1.<token>` | CDN 隐藏 token、避免日志泄露 |

失败立即 `close(code=1008, reason="missing user token")`。握手成功后服务端会**立即下发**一条 `ready` 帧，客户端收到后才能发 `start`。

### 4.2 客户端 → 服务端 `ClientFrame`

仅 3 种控制帧：

```jsonc
// 开始一轮对话；data 必填 ChatRequestPayload
{"type": "start", "data": {
  "session_id": "sess_abc",
  "query": "项目的 ResultValidator 多轮上限怎么配？",
  "agent_mode": true,              // 可选，null 表示沿用 session 默认
  "enable_thinking": false,        // 可选
  "model_preset": "fast",          // 可选
  "max_tool_rounds": 5,            // 可选
  "retrieve_top_k": 8,             // 可选
  "custom_system_prompt": null,    // 可选
  "skip_retrieval": false          // 可选
}}

// 中断当前轮
{"type": "stop"}

// 心跳保活
{"type": "ping"}
```

> **取舍**：v1 不支持 `regenerate` / `branch`；前端如需重生成，**不需要** 删除会话，只要在 WS 上重新发一次 `start` 即可（服务端无状态）。

### 4.3 服务端 → 客户端 `ServerFrame`

`ServerFrame.type` 与业务 `ChatEventType.value` 一一对应，外加 `ready` / `pong` 两个连接级帧：

| frame type | 来源 | 触发时机 | data 字段 |
|---|---|---|---|
| `ready` | 协议层 | 握手成功后立刻下发 | `{subprotocol, user_id}` |
| `pong` | 协议层 | 客户端 `ping` 回应 | `{}` |
| `session.ready` | ChatService | user 消息落库后 | `{session_id, user_message_id, agent_mode, model_preset}` |
| `retrieval.started` | ChatService | 服务端开始初次召回 | `{query, top_k}` |
| `retrieval.done` | ChatService | 召回完成 | `{hit_count, time_ms, chunks: [...]}` |
| `thinking.delta` | StreamAccumulator | 思考链增量 | `{text}` |
| `content.delta` | StreamAccumulator | 正文增量 | `{text}` |
| `tool_call.started` | StreamAccumulator | 模型发起工具调用 | `{index, id, name}` |
| `tool_call.args_delta` | StreamAccumulator | 工具参数流式片段 | `{index, text}` |
| `tool_call.completed` | ChatService | 工具执行完毕 | `{id, name, args, result_brief, items_added, time_ms}` |
| `tool_round.done` | ChatService | 一轮工具批次结束 | `{round, tool_calls: [...]}` |
| `message.done` | ChatService | 一条 assistant 落库 | `{message_id, role, round, finish_reason, tool_calls_count, citations_count, usage}` |
| `turn.done` | ChatService | 整轮结束 | `{rounds, tool_calls_count, time_ms, user_message_id, assistant_message_ids}` |
| `error` | 协议 / ChatService | 任意阶段异常 | `{phase, error, ...extras}` |

### 4.4 协议要点

1. **`thinking.delta` 与 `content.delta` 严格区分**——前端必须在不同区域分别渲染，**不要**在同一缓冲区里 append。
2. **Agent 多轮**：单轮内可能产生多条 `message.done`（每个 LLM 调用产出一条），整轮以 `turn.done` 收尾。
3. **`error` 与 `turn.done` 互不替代**：检索失败之类的局部异常会先发 `error(phase=retrieve)`，主流程继续直到 `turn.done`；只有 LLM 致命异常才会缺失 `turn.done`。
4. **STOP 取消语义**：客户端发 `{type:"stop"}` → 服务端取消当前 chat 任务 → 回发 `error(phase=stop, cancelled=true)`；**已写入的 user 消息保留**，**未完成的 assistant 不落库**（与"用户中断不计费"语义匹配）。
5. **一个 WS = 一个轮窗口**：单连接最多一个进行中的 chat 任务；进行中再发 `start` 会回 `error(phase=start)`；前端如需并发，开新 WS 连接即可。

### 4.5 典型事件序列

RAG 单轮：

```
ready → session.ready → retrieval.started → retrieval.done
      → content.delta × N → message.done → turn.done
```

Agent 一轮工具调用 + 一轮收尾：

```
ready → session.ready → retrieval.started → retrieval.done
      → (round 0: thinking.delta × N → content.delta × N → tool_call.started/args_delta × N
          → tool_call.completed → tool_round.done → message.done)
      → (round 1 收尾: content.delta × N → message.done)
      → turn.done
```

Stop 中断：

```
ready → session.ready → retrieval.done → content.delta × N
      → error(phase=stop, cancelled=true)   # 客户端发 stop 后
```

## 5. REST API 规范

所有 REST 端点都通过 `Depends(get_current_user_id)` 校验请求头 `X-User-Id`，**跨用户访问统一返回 404**（不暴露存在性，与 `Knowledge` 模块风格一致）。

| Method | Path | 说明 | 请求体 / Query |
|---|---|---|---|
| POST   | `/api/chat/sessions`               | 创建会话 | `ChatSessionCreateRequest` |
| GET    | `/api/chat/sessions`               | 我的会话列表（按 `last_message_at DESC`） | `page, page_size` |
| GET    | `/api/chat/sessions/{id}`          | 会话详情 | — |
| PATCH  | `/api/chat/sessions/{id}`          | 重命名 | `ChatSessionRenameRequest({title})` |
| DELETE | `/api/chat/sessions/{id}`          | 软删除（级联软删 MongoDB 消息） | — |
| GET    | `/api/chat/sessions/{id}/messages` | 历史消息（按 `create_time` 正序，分页） | `page, page_size` |

请求 / 响应 schemas 见 `api/schemas/chat/session.py`：

- `ChatSessionCreateRequest`: `title / knowledge_base_ids / model_preset / agent_mode / enable_thinking / max_tool_rounds / system_prompt`
- `ChatSessionInfo`: 完整会话详情（含 `message_count` / `last_message_at` 等运行计数）
- `ChatSessionListResponse`: `{items, total, page, page_size}`
- `ChatMessageListResponse`: 把 `tool_calls / citations / usage` 全部展开，前端可直接渲染

> **取舍**：v1 不暴露 `POST /sessions/{id}/regenerate`。重生成由前端连接 WS 后再发 `start` 实现，服务端无状态。

## 6. 持久化模型

> 目录命名约定：MySQL / MongoDB 各加 `conversation/` 子目录，与 `base / business / extract` 平级。
> 未来 Agent trace、长期 memory 等"对话流"相关表都会落入此目录。

### 6.1 ChatSession（MySQL）

继承 `BaseModel`（自动获得 `status / creator / create_time / updater / update_time / deleted` 6 个标准审计字段）。**不直接继承 `AgentMixin`**，因为本表 `session_id` 主键是 `String(64)` UUID 风格，与 `AgentMixin.session_id` 的 `BigInteger` 类型冲突。

| 字段 | 类型 | 说明 |
|---|---|---|
| `session_id` (PK) | `String(64)` | UUID 风格，与 MongoDB `chat_message.session_id` 一一对应 |
| `user_id` | `String(64)` | 必填，权限校验入口 |
| `title` | `String(255)` | 默认"新会话"，首条消息后由 LLM 异步生成 |
| `knowledge_base_ids` | `JSON` | 本会话允许检索的 KB 列表；空表示用户全量 KB |
| `model_preset` | `String(64)` | 引用 `[llm.presets.<name>]`，默认 `fast` |
| `agent_mode` | `Boolean` | `True` 默认；`False` 走 RAG 单轮快路径 |
| `enable_thinking` | `Boolean` | 是否启用思考链（仅 reasoning 类模型生效） |
| `max_tool_rounds` | `SmallInteger` | Agent 模式下含工具批次的轮数上限，默认 5 |
| `system_prompt` | `Text` | 用户自定义 system prompt；NULL 时用模块默认值 |
| `message_count` | `Integer` | 已落 MongoDB 的消息总数（user + assistant + tool 全计） |
| `last_message_at` | `DateTime` | 最近活跃时间（用于会话列表倒序） |

复合索引：

- `idx_user_lastmsg(user_id, last_message_at)` — "我的会话列表"按活跃度倒序
- `idx_user_deleted(user_id, deleted)` — 软删除过滤

### 6.2 ChatMessage（MongoDB / Beanie）

继承 `BaseDocument`（沿用相同审计字段名 + `soft_delete()` / `restore()` 方法），主键采用项目里一致的 `chatmsg_<uuid>` 字符串风格。

```python
class ChatRole(str, Enum):
    SYSTEM = "system"; USER = "user"; ASSISTANT = "assistant"; TOOL = "tool"

class Citation(PydanticBaseModel):
    chunk_id: str; document_id: Optional[str]; knowledge_base_id: Optional[str]; score: float

class ToolCallRecord(PydanticBaseModel):
    id: str; name: str; arguments: dict; result_brief: Optional[str]; items_added: int

class TokenUsageRecord(PydanticBaseModel):
    prompt_tokens: int; completion_tokens: int
    thinking_tokens: Optional[int]; total_tokens: int

class ChatMessage(BaseDocument):
    id: str = Field(..., alias="_id")          # chatmsg_<uuid>
    session_id: str                            # 与 MySQL.chat_session.session_id 一致
    user_id: str                               # 冗余存储，便于跨会话审计
    role: Literal["system","user","assistant","tool"]
    content: str = ""
    # assistant 专属
    thinking: Optional[str] = None
    tool_calls: List[ToolCallRecord] = []
    citations: List[Citation] = []
    usage: Optional[TokenUsageRecord] = None
    finish_reason: Optional[str] = None
    # tool 专属
    tool_call_id: Optional[str] = None         # 关联同会话最近 assistant.tool_calls[*].id
    metadata: Dict[str, Any] = {}              # model 名 / provider / event_id / AB 标记 等
```

3 个索引：`(session_id, create_time)` / `(user_id, create_time desc)` / `(deleted, create_time desc)`。

### 6.3 跨库映射

通过 `session_id` 字符串关联，**无外键**；`chat_session.message_count` 作为冗余计数由 `chat_session_repo.touch(delta=N)` 在写消息后**一次性原子刷新**（避免每条消息触发一次 MySQL update）。

### 6.4 持久化时机

| 触发点 | 持久化动作 |
|---|---|
| 进入 `ChatService.chat_stream()` | user 消息立刻落 MongoDB（保证客户端断连不丢失） |
| 每个 LLM 调用结束 | 整体写入一条 assistant 消息（避免每 delta 一次 I/O） |
| 每次工具执行完成 | 写入一条 `role=tool` 消息（带 `tool_call_id` 关联 assistant） |
| 整轮结束 | `chat_session_repo.touch(delta=N)` 一次性更新 MySQL 计数 |

## 7. ChatService 编排流程

`ChatService.chat_stream(request) -> AsyncIterator[ChatEvent]` 是单一入口，让上层（WS 路由）零业务侵入地把 `ChatEvent` 翻译为 WS 帧；背压自然透传（上层写入慢 → 反压到 LLM 拉流）。

### 7.1 核心流程图

```
┌────────────────────────────────────────────────────────────────────┐
│                         ChatService.chat_stream                     │
│                       (async generator: ChatEvent)                  │
├────────────────────────────────────────────────────────────────────┤
│  1) get_session(权限校验)  ─→  SESSION_READY                       │
│                                                                     │
│  2) 持久化 user 消息  ─────→  MongoDB                              │
│                                                                     │
│  3) RetrieveService.retrieve()                                      │
│     ├ RETRIEVAL_STARTED                                             │
│     └ RETRIEVAL_DONE  (含 hits 预览 + 耗时)                         │
│                                                                     │
│  4) 历史收敛：drop_dangling → window → token_window → summary      │
│                                                                     │
│  5) compose_chat_messages(system, history, chunks, user)            │
│                                                                     │
│  6) 主循环（max_rounds 轮 + 1 收尾轮）                              │
│      for round in range(max_rounds):                                │
│          ├ astream → StreamAccumulator → CONTENT/THINKING/TOOL_*    │
│          ├ resp = acc.finalize()                                    │
│          ├ 持久化 assistant + 透出 MESSAGE_DONE                     │
│          ├ if not resp.tool_calls: return                           │
│          ├ asyncio.gather(并行执行 tool_calls)                       │
│          ├ 每条 tool 持久化 + 透出 TOOL_CALL_COMPLETED              │
│          └ 透出 TOOL_ROUND_DONE                                     │
│      若仍含 tool_calls → 收尾轮 astream(tools=None) → MESSAGE_DONE  │
│                                                                     │
│  7) touch_session(delta=新增条数)  ─→  MySQL                       │
│                                                                     │
│  8) 首轮（message_count==0）→ TitleService.schedule_in_background   │
│                                                                     │
│  9) TURN_DONE  (汇总 rounds / tool_rounds / tool_calls_count /      │
│                 citations_count / 各阶段耗时 / error)                │
└────────────────────────────────────────────────────────────────────┘
```

### 7.2 关键设计

#### 7.2.1 RAG / Agent 路径统一

`_run_loop_real` 通过 `tools_schema is None` 区分两条路径，**收尾轮逻辑共用**。RAG 模式 `max_rounds=1` 且模型本不应返回 `tool_calls`，万一返回则记 warning 并按"不再继续"处理。

#### 7.2.2 citations 合并语义

| 阶段 | citations 含义 |
|---|---|
| 第 0 轮 assistant | 种子检索命中 + 本轮工具新增（去重） |
| 后续每轮 assistant | 仅本轮工具新增 |
| `TURN_DONE.citations_count` | 全轮去重后的 chunk_id 总数 |

这样每条 assistant 都自带"它当时引用了什么"，前端可在每条气泡下方独立渲染引用块。

#### 7.2.3 失败隔离粒度

| 失败点 | 处理 |
|---|---|
| 加载 session 失败 / session 不存在 | `ERROR(load_session)` 立即返回 |
| 检索失败 | `ERROR(retrieve)` + 按"空命中"继续 RAG |
| 工具执行失败 | `KnowledgeNavToolKit.call` 已封装为 error string，照常进入下一轮 LLM |
| LLM 流式调用失败 | `ERROR(llm_stream)` 中断主循环但仍透 `TURN_DONE` |

#### 7.2.4 长上下文兜底（三段式收敛）

依赖 `src/prompts/chat/history_compressor.py`：

```python
history = drop_assistant_tool_dangling(history)
history = apply_history_window(history, max_turns=12)        # 轮次滑窗
history = apply_token_window(history, max_tokens=8000, ...)  # token 滑窗（按对话轮整轮回退）
if estimate_history_tokens(history) > soft_limit:
    summary_dict, history = await compress_history_to_summary(
        history, summarize_fn=fast_summarize, keep_recent_turns=4,
    )
```

要点：

- 历史滑窗以"对话轮"为最小粒度，`assistant.tool_calls + role=tool` **整轮保留或整轮丢弃**，避免出现 LiteLLM 不接受的"孤儿 tool"。
- Token 估算优先调 `litellm.token_counter(model, messages, tools)`，失败兜底用经验值（中文 ≈ 1.6 chars/token，英文 ≈ 4 chars/token），未知模型不抛异常。
- 摘要压缩通过 `SummarizeFn` 由 ChatService 注入（典型用 `fast` preset 走 `agenerate`），回调异常自动退化为 `(None, recent_part)`，**摘要消息不持久化**，仅作为运行期 `messages` dict 注入。

#### 7.2.5 首轮异步起标题

通过 `session.message_count == 0` 判断；`TitleService.schedule_in_background` 返回 `asyncio.Task`（task name = `title:<session_id>` 便于日志），主流程**不 await**，下一次 `list_sessions` 拉到即可。失败兜底为 query 前 20 字。

#### 7.2.6 LLMClient 缓存

按 `model_preset` 名缓存，直接调 `create_llm_client_from_preset(preset)`。**Chat 不走 `components.json`**——见第 9 节配置说明。

### 7.3 与 ResultValidator 的关系

| 维度 | ResultValidator | ChatService Agent |
|---|---|---|
| 触发对象 | 单次 query 的检索结果 | 用户多轮对话 |
| LLM 调用 | `agenerate`（非流式） | `astream`（流式） |
| 收尾输出 | `[验证状态]` 标签 | 自然语言回答 + citations |
| 持久化 | 不持久化 | 整体写入 MongoDB |
| 工具集 | `expand_context / drill_down / roll_up / skeleton / re_retrieve` | `expand_context / drill_down / skeleton`（**子集**） |

两者共享 `KnowledgeNavToolKit` 基类，`ValidatorToolKit` 通过钩子追加 `re_retrieve`。

## 8. Agent 工具集

### 8.1 工具白名单（Chat 默认）

| 工具名 | 入参 | 调用底层 | 行为 |
|---|---|---|---|
| `expand_context` | `chunk_id, window_size` | `capabilities/navigation/ContextWindow` | 取目标 chunk 前后窗口 |
| `drill_down` | `section_id?, document_id?` | `capabilities/navigation/DrillDown` | 取 section / document 下的所有 chunks |
| `skeleton` | `document_id` | `capabilities/navigation/Skeleton` | 取文档目录骨架（不展开正文） |

> **不暴露** `roll_up`（"chunk 上溯到 section 标题"在对话场景下价值较低，可用 `skeleton` 替代）和 `re_retrieve`（避免 LLM 直接决定扩大召回范围）。

### 8.2 KnowledgeNavToolKit

```python
class KnowledgeNavToolKit:
    def __init__(self, supplemented_items, *, enabled_tools=DEFAULT_NAV_TOOLS): ...
    def schemas(self) -> List[ToolSchema]: ...
    def has(self, name: str) -> bool: ...
    async def call(self, name: str, args: dict) -> str: ...

    # 子类钩子（ValidatorToolKit 通过这两个钩子追加 re_retrieve）
    def _register_extra_tools(self, registry): ...
    def _extra_tool_schemas(self) -> List[ToolSchema]: ...
```

工具结果以**自然语言文本**返回（参考 `tools.py::format_chunks_for_llm`），同时把新拉到的 chunks 追加到 `supplemented_items`，最终由 `ChatService` 合入 citations。

### 8.3 工具异常的处理约定

`KnowledgeNavToolKit.call` 内部对所有异常做了 try/except 兜底，返回形如 `"[tool_error] ..."` 的字符串。这样：

- LLM 在下一轮看到 error string 后通常会自我纠正（换工具 / 换参数）；
- 主流程**不会因为工具崩溃中断**，仍会走完 `tool_round.done → message.done`；
- 异常细节会通过 `loguru` 打到服务端日志，便于排障。

## 9. 配置项（`config/config.toml [chat]`）

> **重要边界**：`config/components.json` 只服务于 **RAG 抽取 Pipeline**——每个组件对应一个 Kafka Worker（见 `src/db/kafka/workers/*Worker`），可独立水平扩容。
>
> Chat 是在线请求路径，**不在 components 体系内**：LLM 选型通过 `[chat]` 节直接引用 `[llm.presets.*]`，无中间层。

### 9.1 配置概念对照

| 配置文件 | 服务对象 | 触发方式 | 扩容轴 |
|---|---|---|---|
| `components.json` | RAG 抽取 Pipeline（Kafka Worker） | Kafka 消息驱动 | Kafka lag → Pod 数 |
| `config.toml [chat]` | Chat 在线服务（ChatService / TitleService） | WS / HTTP 请求 | QPS → API 节点数 |

### 9.2 `[chat]` 节字段总览

```toml
[chat]
agent_model_preset = "fast"          # 主对话 LLM 的 preset（兜底）
title_model_preset = "fast"          # 异步起标题用的 preset（性价比优先）
default_agent_mode = true            # 默认是否启用 Agent 工具循环
default_enable_thinking = false      # 默认是否启用思考链
default_max_tool_rounds = 5          # Agent 模式默认工具批次上限
thinking_budget = 4096               # enable_thinking=true 时透传 LLMClient.astream
max_completion_tokens = 0            # 0 / 缺省 = 由模型 / preset 决定

[chat.retrieval]
top_k = 8                            # 服务端初始检索 top_k

[chat.history]
max_messages = 40                    # 加载历史消息条数上限
max_turns = 12                       # 轮数滑窗（与 token 滑窗叠加）
max_tokens = 8000                    # token 滑窗预算
summary_compress_threshold_turns = 15  # 超此轮数触发摘要压缩；0 = 关闭
summary_keep_recent_turns = 4          # 摘要压缩后保留的最近轮数
```

加载方式：`ChatServiceConfig.from_config_manager()`（缺失字段沿用类默认；`ConfigManager` 异常时退化为全部默认）。WS 路由 `_get_chat_service()` 已自动装配。

### 9.3 LLM 客户端装配

`ChatService._get_llm_client(preset)` 与 `TitleService._get_client()` 都直接调 `create_llm_client_from_preset(preset)`，按 preset 名缓存，**不读 `components.json`**。

需要为 chat 细调模型时，运维在 `config.toml` 加一个自定义 preset 即可：

```toml
[llm.presets.chat_reasoning]
model = "deepseek/deepseek-reasoner"
temperature = 0.1
max_tokens = 4096
thinking_budget = 4096
```

然后把 `[chat].agent_model_preset = "chat_reasoning"`，重启服务即可生效；代码层 0 改动，也不会污染 `components.json` 的语义。

### 9.4 拆分 `agent_model_preset` / `title_model_preset` 的理由

主对话和起标题用不同模型是常见的成本优化（reasoning 模型贵 → 起标题用 fast；主对话 reasoning）；两个独立字段语义最清楚，且对运维而言"改谁不影响谁"。

## 10. 前端集成指南

### 10.1 一次完整对话的最小流程

```
1. POST /api/chat/sessions                    → 拿到 session_id
2. WS  /api/chat/ws?token=<user_id>           → 等 ready 帧
3. WS  send {"type":"start", "data":{...}}    → 进入实时事件流
4. 监听各类 ServerFrame：
   - retrieval.* → 展示"正在检索 / 命中 N 段"
   - thinking.delta → 思考区域实时拼接
   - content.delta  → 正文区域实时拼接
   - tool_call.* → 时间线展示工具调用
   - message.done → 一条 assistant 收尾（可固化引用块）
   - turn.done    → 整轮收尾（可解锁输入框）
   - error        → 局部异常提示 / 致命异常关闭对话
5. 用户中断：WS send {"type":"stop"}
6. 复用：跳到第 3 步发新 start（无需新建 session）
```

### 10.2 浏览器 WebSocket 样板

```js
const ws = new WebSocket(
  "ws://127.0.0.1:8000/api/chat/ws?token=<user_id>",
  ["aks-chat-v1"],
);

ws.onmessage = (e) => {
  const frame = JSON.parse(e.data);
  switch (frame.type) {
    case "ready":            /* 握手完成，可发 start */ break;
    case "session.ready":    /* user 消息已落库 */ break;
    case "retrieval.started":/* 检索开始 */ break;
    case "retrieval.done":   /* 命中 chunks */ break;
    case "thinking.delta":   /* 思考链增量，独立缓冲区渲染 */ break;
    case "content.delta":    /* 正文增量 */ break;
    case "tool_call.started":     /* 模型发起工具调用 */ break;
    case "tool_call.args_delta":  /* 工具参数流式 */ break;
    case "tool_call.completed":   /* 工具执行完毕 */ break;
    case "tool_round.done":  /* 一轮工具批次结束 */ break;
    case "message.done":     /* 一条 assistant 落库 */ break;
    case "turn.done":        /* 整轮结束 */ break;
    case "error":            /* 局部 / 致命异常 */ break;
  }
};

// 收到 ready 后再发 start
function startTurn(sessionId, query) {
  ws.send(JSON.stringify({
    type: "start",
    data: { session_id: sessionId, query, agent_mode: true },
  }));
}

// 中途想停：
function stop() {
  ws.send(JSON.stringify({ type: "stop" }));
}
```

### 10.3 调试 / 自测 driver

仓库自带 `test/e2e/chat_client.py`（不依赖前端，纯命令行）：

```bash
# Terminal A：启动服务
uv run uvicorn main:app --reload --host 0.0.0.0 --port 8000

# Terminal B：跑客户端
uv run python test/e2e/chat_client.py --mode rag    # RAG 单轮
uv run python test/e2e/chat_client.py --mode agent  # Agent 多轮
uv run python test/e2e/chat_client.py --mode stop --stop-after 2.0  # 中断
```

driver 会自动从 MySQL `knowledge_base` + Milvus `chunk` 探测一个"有 chunk 索引"的可用知识库，建会话、连 WS、打字机式打印事件流，并把完整事件流落到 `logs/e2e/<时间戳>_<mode>/events.jsonl`。

## 11. 关键设计取舍（决策记录）

| # | 议题 | 选定方案 | 备选方案 | 选型理由 |
|---|---|---|---|---|
| 1 | 主流式协议 | **WebSocket** | SSE / HTTP chunked | 双向通信，便于 stop / regenerate / 中途回填工具结果；前端实现简单 |
| 2 | 工作模式 | Agent 优先 + RAG 单轮快路径 | 只做 RAG / 只做 Agent | 两类场景都要；`agent_mode=false` 时跳过工具循环走快路径 |
| 3 | 服务端无条件初次召回 | 每条 user 输入都强制检索一次 | 让 LLM 决定是否检索 | 保证最低召回保障；LLM 仅决定"是否进一步细化" |
| 4 | Agent 工具白名单 | `expand_context / drill_down / skeleton` | + `re_retrieve / roll_up` | 收敛"扩大召回"决策权到服务端，行为可预测、可审计 |
| 5 | 持久化分库 | MySQL 元信息 + MongoDB 消息内容 | 全 MySQL / 全 MongoDB | 元信息结构化、消息半结构化；冗余 `message_count` 由 `touch()` 一次刷新 |
| 6 | 思考链与正文独立 channel | `thinking.delta` ≠ `content.delta` | 同一缓冲区按标记分割 | 前端必须分区域渲染；事件流自带语义比解析正文标记更稳定 |
| 7 | Stop 取消语义 | 已写 user 保留；未完 assistant 不落库 | 强制保留 partial assistant | "用户中断不计费"语义；避免半截内容污染历史 |
| 8 | 一个 WS = 一个轮窗口 | 进行中再发 `start` 拒绝 | WS 内多任务复用 | 后端无需维护流多路复用复杂度；前端如需并发开新连接 |
| 9 | 重生成无独立端点 | 直接在 WS 发新 `start` | 单独 `POST /regenerate` | 服务端无状态；少一个端点意味着少一处历史一致性风险 |
| 10 | WS 鉴权双通道 | query token + 子协议 token | 仅 cookie / 仅 header | 浏览器原生 WS 不能设 header；CDN 场景需要子协议藏 token |
| 11 | 跨用户访问统一 404 | 不暴露存在性 | 403 | 与 `Knowledge` 模块风格一致；防枚举 |
| 12 | LLM 配置位置 | `config.toml [chat]` 引用 `[llm.presets.*]` | 把 chat 当成 components 写进 `components.json` | `components.json` 仅服务于 Kafka Worker；chat 是在线路径，强行做成组件会破坏 components 体系语义 |
| 13 | 主对话 / 标题 preset 拆开 | `agent_model_preset` + `title_model_preset` | 共用一个 preset | 主对话用 reasoning、起标题用 fast 是常见的成本优化；两字段对运维"改谁不影响谁" |
| 14 | 历史滑窗最小粒度 | "对话轮"整轮保留或整轮丢弃 | 单条消息 | 避免出现 LiteLLM 不接受的"孤儿 tool"（assistant.tool_calls 与 role=tool 必须配对） |
| 15 | 摘要压缩接入方式 | 注入式 `SummarizeFn` 回调 | 模块内硬编码调 LLM | 解耦 prompts 模块与 LLM 客户端；回调异常可优雅退化 |
| 16 | 工具集合公共基类 | `KnowledgeNavToolKit` + 子类钩子 | Chat / Validator 各维护一份 | 避免双份维护；`ValidatorToolKit = ToolKit` 兼容别名让 `result_validator.py` 零改动 |
| 17 | citations 写入粒度 | 每条 assistant 自带"它当时引用了什么" | 整轮汇总到 `turn.done` | 前端可在每条气泡下渲染独立引用块；溯源更精准 |

## 12. 与 RAG 抽取 Pipeline 的边界

经常被混淆的一对概念，明确如下：

| 维度 | RAG 抽取 Pipeline | Chat 在线服务 |
|---|---|---|
| 入口 | Kafka 消息（如 `db_write.embedding.start`） | WS / HTTP 请求 |
| 调度对象 | `BaseWorker` 子类（FileParser / TextSplitter / FileSummary / KGExtractor / ...） | `ChatService.chat_stream` async generator |
| 配置文件 | `config/components.json` | `config/config.toml [chat]` |
| 扩容轴 | Kafka lag → Pod 数（独立 scaling profile） | API 节点数（QPS） |
| LLM 选型 | `components.json` 里 `llm_preset` / 内联 `llm` 字段 | `[chat].agent_model_preset` / `title_model_preset` 引用 `[llm.presets.*]` |
| 失败语义 | Kafka 重试 / 死信队列 | `error` 事件 + 主流程降级继续 |

**一句话总结**：Pipeline 是离线 / 后台、Worker 维度横向扩容；Chat 是在线 / 同步、API 维度横向扩容。两者**不共享 components 配置**。

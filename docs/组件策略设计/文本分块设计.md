# 文本分块策略 — 技术设计文档

> 模块路径：`src/index/common_file_extract/splitter/`
> 服务编排：`src/service/knowledge/components/text_splitter_service.py`
> 更新日期：2026-02-27

---

## 1. 核心设计理念

本模块采用 **"结构优先，长度兜底"（Structure-First, Size-Constraint）** 的设计哲学：

1. **结构优先**：切分时优先尊重文档的自然结构（段落边界 `\n\n`），绝不在段落内部硬切。
2. **长度兜底**：只有当单个段落本身超过 `chunk_size` 时，才启动递归细分。
3. **异构独立**：文本、图片、表格三种元素类型各自独立处理，互不干扰。

## 4. 元素分流处理策略

### 4.1 处理模型：文本累积 buffer

采用 **文本累积 buffer** 模型。遍历 `ParseResult.elements` 列表时，连续的文本 Element 会先累积到 buffer 中，遇到非文本元素（图片/表格/标题）或循环结束时再统一 flush（合并 + 切分）。这避免了连续短文本 Element 各自产生碎片 chunk 的问题。

```
Element 列表（按文档顺序）:

  [标题A] → flush buffer → 创建 Section，记录 section_id
      ↓
  [文本1] → 清洗后追加到 buffer
      ↓
  [文本2] → 清洗后追加到 buffer
      ↓
  [图片1] → flush buffer（文本1+文本2 合并切分）→ 创建 Image Chunk
      ↓
  [文本3] → 清洗后追加到 buffer
      ↓
  [表格1] → flush buffer（文本3 切分）→ 组装 + 切分表格
      ↓
  [文本4] → 清洗后追加到 buffer
      ↓
  [文本5] → 清洗后追加到 buffer
      ↓
  [标题B] → flush buffer（文本4+文本5 合并切分）→ 创建新 Section
      ↓
  [文本6] → 清洗后追加到 buffer
      ↓
  (循环结束) → flush buffer（文本6 切分）
```

**buffer 状态**：
- `text_buffer: List[str]`：累积的已清洗文本列表
- `buffer_element_ids: List[str]`：累积的 Element ID（用于溯源）
- `buffer_page_index: Optional[int]`：首个文本元素的页码

**flush 触发时机**：标题元素、图片元素、表格元素、循环结束

**flush 逻辑**：
1. 用 `"\n\n"` 拼接 buffer 中的所有文本（自然喂给 structure_first 的段落拆分）
2. 调用 `text_splitter.split_text()` 切分合并后的文本
3. 生成 Chunk，`element_ids` 包含所有参与合并的 Element ID
4. 清空 buffer

关键特性：
- **标题**先 flush buffer 再创建 Section，确保前文文本不会跨章节合并
- **连续文本**会被累积并合并切分，避免碎片化短 chunk
- **图片/表格**先 flush buffer 再独立处理，保证文本流不被截断后丢失
- **溯源**：合并后的 Chunk 的 `element_ids` 包含所有参与合并的 Element ID

### 4.2 Section 关联机制

```
                 section_id 状态机
                 ═══════════════

遇到标题元素 → current_section_id = 新创建的 section.section_id
遇到其他元素 → 使用 current_section_id 关联

效果：
  [标题A]  → section_id = "section-aaa"
  [文本1]  → chunk.section_id = "section-aaa"
  [图片1]  → chunk.section_id = "section-aaa"
  [标题B]  → section_id = "section-bbb"
  [文本2]  → chunk.section_id = "section-bbb"
```

最后在 `split_document()` 末尾，会反向填充每个 Section 的 `chunk_id_list`。

## 5. 文本清洗流水线（TextCleaner）

文本在切分前经过 4 步流水线式清洗（**顺序不可调整**）：

```
原始文本
  │
  ▼
┌──────────────────────────────────────────────────────────┐
│ Step 1: clean_special_chars()                            │
│   - 移除控制字符 (\x00-\x08, \x7F-\x9F 等)             │
│   - 移除零宽度字符 (\u200B-\u200F, \u2060-\u206F 等)    │
│   - 统一换行符 (\r\n → \n, \r → \n)                     │
└──────────────────────┬───────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────┐
│ Step 2: clean_text()                                     │
│   - 行内水平空白归一化 (多空格/制表符 → 单空格)          │
│     正则: [^\S\n]+ → ' ' (不影响换行符)                  │
│   - 移除重复标点 (!!! → !, ... → . 等)                   │
│   - 段落边界规范化 (\n{3,} → \n\n)                       │
└──────────────────────┬───────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────┐
│ Step 3: normalize_whitespace()                           │
│   - 逐行 strip() 去除行首尾空格                          │
│   - 连续空行合并为单个空行 (保留段落 \n\n 边界)          │
│   - 去除首尾空行                                         │
└──────────────────────┬───────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────┐
│ Step 4: remove_extra_newlines()                          │
│   - 限制最大连续换行符数量 (默认 max=2)                  │
└──────────────────────┬───────────────────────────────────┘
                       ▼
                  清洗后文本
```

**设计目标**：在清除噪声的同时，严格保留 `\n\n` 段落边界，为后续结构切分提供干净的输入。

## 6. 文本切分策略（TextSplitter）

提供 5 种切分算法，通过 `SplitConfig.split_method` 选择：

| 方法 | 枚举值 | 计量单位 | 适用场景 | 推荐度 |
|------|--------|---------|---------|-------|
| 两阶段结构切分 | `structure_first` | 字符 | 所有类型文档 | ⭐⭐⭐⭐⭐ |
| 递归切分 | `recursive` | 字符 | 通用文档 | ⭐⭐⭐⭐ |
| 常规切分 | `regular` | 字符 | 格式规整文档 | ⭐⭐⭐ |
| 语义切分 | `semantic` | Token | 高精度检索 | ⭐⭐⭐ |
| Token切分 | `token` | Token | 特定 LLM 限制 | ⭐⭐⭐ |

### 6.1 两阶段结构切分（structure_first）⭐ 默认推荐

核心算法，分两个阶段工作：

```
                    输入文本
                      │
          ┌───────────┴───────────┐
          │  阶段1：宏观拆分       │
          │  re.split(r'\n{2,}')  │
          │  按段落边界拆分        │
          └───────────┬───────────┘
                      │
              ┌───────┴───────┐
              ↓               ↓
      [段落1, 段落2, 段落3, 段落4, ...]
              │
          ┌───┴───────────────────────────────────┐
          │  阶段2：逐段落微观决策                  │
          └───┬───────────────────────────────────┘
              │
              ↓ 遍历每个段落
              │
         len(段落) > chunk_size ?
          ╱                 ╲
        是                   否
        ↓                    ↓
   ┌─────────────┐    buffer + 段落 <= chunk_size ?
   │ 先提交 buffer│      ╱              ╲
   │ 再用递归切分 │    是                 否
   │ 器切碎该段落 │    ↓                  ↓
   │ (分隔符优先级│  合并到 buffer      提交 buffer
   │  \n>。>，> ) │  buffer += \n\n    段落成为新 buffer
   └─────────────┘  + 段落
              │
              ↓
         最终提交剩余 buffer
              │
              ↓
          输出 chunks[]
```

**算法特点**：
- 绝不在段落内部切割（除非段落本身 > `chunk_size`）
- 小段落贪婪合并，避免产生碎片 chunk
- 超大段落使用 `RecursiveCharacterTextSplitter` 递归细分，分隔符优先级：`\n` → `。` → `，` → 空格
- 建议 `chunk_overlap = 0`，段落边界天然保证语义连续性

### 6.2 递归切分（recursive）

直接调用 LangChain 的 `RecursiveCharacterTextSplitter`：
- 多级分隔符递归：`\n\n` → `\n` → 空格 → 硬切
- 支持 `chunk_overlap`（推荐 chunk_size 的 10%-20%）
- 不做段落级别的贪婪合并

### 6.3 常规切分（regular）

调用 LangChain 的 `CharacterTextSplitter`：
- 固定单一分隔符（默认 `\n`）
- 适合格式规整、行结构明确的文档

### 6.4 语义切分（semantic）

调用 `SentenceTransformersTokenTextSplitter`：
- 基于 Sentence Transformer 嵌入模型（默认 `all-MiniLM-L6-v2`）
- `chunk_size` 单位是 **token 数**，不是字符数
- 适合对检索精度要求极高的场景

### 6.5 Token 切分（token）

调用 LangChain 的 `TokenTextSplitter`：
- 按特定 tokenizer 编码计算长度（默认 `cl100k_base` 即 GPT-4 tokenizer）
- `chunk_size` 单位是 **token 数**
- 适合需要精确控制 LLM 上下文窗口的场景

## 7. 表格切分策略（TableSplitter）

### 7.1 表格组装

将表格的三个组成部分组装为标准格式文本：

```
table_caption: 表1：2025年销售数据
table_body: <table>...<tr>...</tr>...</table>
table_footnote: 数据来源：内部销售系统
```

### 7.2 切分决策

```
         组装完整表格
              │
              ↓
     len(完整表格) <= 4000 字符 ?
          ╱          ╲
        是             否
        ↓              ↓
   返回完整表格    进入超长表格切分
   [single chunk]      │
                       ↓
                  split_large_table()
```

### 7.3 超长表格按行切分 + 表头冻结

```
┌─────────────────────────────────────────────────────┐
│  1. 解析表格行 (_parse_table_rows)                   │
│     支持三种格式:                                    │
│     - HTML: 按 <tr> 标签正则匹配                     │
│     - Markdown: 按含 | 的行分割                      │
│     - 纯文本: 按换行符分割                           │
│                                                      │
│  2. 计算固定开销                                     │
│     fixed = len(caption) + len(footnote)             │
│           + len(header_row) + 50 (格式字符)          │
│     available = chunk_size - fixed                   │
│                                                      │
│  3. 按行累积切分                                     │
│     每个批次强制以 header_row 开头                    │
│     累积行直到接近 available 上限                     │
│     满了就封装为一个切片，开启新批次                  │
│                                                      │
│  4. 每个切片都包含完整的:                            │
│     caption + header + 数据行 + footnote             │
└─────────────────────────────────────────────────────┘
```

**效果示意**（原表 100 行，按行切分为 3 片）：

```
┌────────────────────────────┐  ┌────────────────────────────┐  ┌────────────────────────────┐
│ Chunk 1                    │  │ Chunk 2                    │  │ Chunk 3                    │
│ caption: 表1：销售数据     │  │ caption: 表1：销售数据     │  │ caption: 表1：销售数据     │
│ header:  日期|产品|销量    │  │ header:  日期|产品|销量    │  │ header:  日期|产品|销量    │
│ rows:    1-35              │  │ rows:    36-70             │  │ rows:    71-100            │
│ footnote: 数据来源:...     │  │ footnote: 数据来源:...     │  │ footnote: 数据来源:...     │
└────────────────────────────┘  └────────────────────────────┘  └────────────────────────────┘
         ↑ 每个切片都是语义自洽的完整表格 ↑
```

## 8. 元素处理器（ElementProcessor）

纯粹的工厂类，提供 4 个静态方法完成 `ElementInfo` → `ChunkInfo / SectionInfo` 的转换：

| 方法 | 输入类型 | 输出类型 | 说明 |
|------|---------|---------|------|
| `create_section_from_element()` | 标题元素 | `SectionInfo` | 提取 level、content、page_index |
| `create_text_chunks()` | 文本元素 + 切分文本列表 | `List[ChunkInfo(TEXT)]` | 每段切分文本创建一个 Chunk |
| `create_image_chunk()` | 图片元素 | `ChunkInfo(IMAGE)` | 继承 MinIO 存储信息 |
| `create_table_chunks()` | 表格元素 + 组装文本列表 | `List[ChunkInfo(TABLE)]` | 保留 table_body/caption/footnote |

所有 Chunk 都携带：
- `element_ids`：溯源到原始 Element
- `section_id`：关联到所属 Section
- `document_id`：关联到文档

## 9. 配置模型（SplitConfig）

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `split_method` | `structure_first` | 切分算法 |
| `chunk_size` | `1000` | 目标 Chunk 大小（字符/Token） |
| `chunk_overlap` | `0` | Chunk 重叠大小 |
| `separators` | `["\n", "。", "，", " "]` | 递归切分的分隔符列表 |
| `model_name` | `all-MiniLM-L6-v2` | 语义切分模型（仅 `semantic` 方法使用） |
| `encoding_name` | `cl100k_base` | Token 编码方式（仅 `token` 方法使用） |
| `enable_text_clean` | `True` | 是否启用文本清洗 |

推荐配置：

| 场景 | chunk_size | chunk_overlap | 说明 |
|------|-----------|---------------|------|
| 通用文档 | 1000 | 0 | 默认配置 |
| 长文档/书籍 | 1500 | 0 | 更大 chunk 保留上下文 |
| 表格密集型 | 2000 | 0 | 表格占比大的文档 |
| 短问答文档 | 300-500 | 0 | 保证问答对完整 |

## 10. 数据模型

```
SplitResult
├── sections: List[SectionInfo]      # 章节/标题列表
```

```
SplitResult
    │
```

## 11. 已知局限与待优化项

### 11.1 标题未嵌入文本流

标题元素只创建 Section，不会将标题文本注入到后续文本 Chunk 的内容中。检索时 Chunk 本身不包含标题上下文，需依赖 `section_id` 关联查询。


### 11.2 代码块处理路径缺失

`ElementProcessor` 已有 `create_code_chunk()` 方法，`ChunkType` 已有 `CODE_BLOCK` 类型，但 `TextSplitterService.split_document()` 的元素遍历中缺少代码块的处理分支。
